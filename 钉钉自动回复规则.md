## 二、消息过滤规则（优先级从高到低）

### 技术性静默（不发送任何消息）

| 规则 | 条件 | 说明 |
|------|------|------|
| 群消息未触发机器人 | 群聊没有 @ 机器人 | 避免机器人干扰群内所有普通对话 |
| 重复回调 | 同一条钉钉消息已经处理 | 防止重复回复 |
| 过期消息 | 超出消息处理时间窗口 | 防止重启后补发旧消息 |

除以上三种技术性情况外，机器人收到并处理的消息都必须给出回复，不能静默跳过。

### 发送者身份

| 规则 | 条件 | 说明 |
|------|------|------|
| 不按员工身份静默 | 总部人员或门店人员触发机器人 | 一律进入正常判断，员工身份不能成为无回复原因 |

### 冷却机制

| 规则 | 时长 | 说明 |
|------|------|------|
| SKIP 冷却 | 5 分钟 | AI 判定 SKIP 的消息 5 分钟后可重新评估 |
| 已回复 | 永久 | 已成功回复的消息不再处理 |
| 记录清理 | 30 分钟 | 状态文件定期清理过期记录 |

## 三、AI 回复规则

### 回复范围：产品问题与日常闲聊

- 日常问候、客气话及普通闲聊允许回复，由模型生成自然、简短的答复。
- 对“你好 / 在吗 / 早上好”等纯问候，不作纯闲聊，固定引导到业务：说明可解答酷太产品尺寸、安装、适配、材质及产品/安装示意图，并请对方直接说明产品名称和问题。
- 产品事实、尺寸、安装、选型、兼容关系和介绍话术统一依据本地 LLM Wiki 回答；禁止因单一产品问题新增本地硬编码回复。Wiki 无可靠证据时进入 REVIEW，不允许模型按常识补全。
- **事实检索硬分层**：安装、尺寸、极限、产品结构、材质、承重、轨道、兼容和公式问题只检索 `fact` 结构化产品卡与安装规则；营销话术、设计建议、空间方案、应用名和竞品内容不得进入同一召回池。事实结论确定后，才允许用话术做不改变结论的简短润色。
- 订单、物流、售后责任、价格审批、内部流程等不能可靠自动处理的正事，不回答具体结论，统一回复：`这个问题请联系人工处理。`

| 可以回复 | 不予回复 |
|----------|----------|
| 产品规格/尺寸/材质/颜色 | 物流/发货/快递查询 |
| 收纳方案设计/选品建议/空间规划 | 订单进度/修改/下单流程 |
| 安装方式/定制范围/轨道选型 | 客资跟进/退回/销售流程 |
| 安装教程/安装说明书：有明确说明书就发送，找不到就回微信视频号 | 孔位图资料不明确时直接猜测 |
| 实物图/产品图/参考图/效果图：资料库有图就直接发送图片 | 把明确要图片的问题改成产品功能解释 |
| 产品对比（魔法抽vs尚酷抽等） | 财务/充值/合同/发票 |
| 竞品对比（酷太vs悍高/诺米等） | 售后工单/安装预约/投诉 |
| | 已安装后的现场异常/摩擦柜门/运行卡滞/左右微调 |
| | 换产品/换整机/换新/退换货 |
| | 价格/折扣/优惠、上架/下架/上市/商城入口 |
| | 人员安排/排班/内部管理 |

### 转人工的情况

- 非产品话题（物流/订单/客资/财务/售后/管理/价格/上架）
- 对方在跟其他人对话（不是问大和）
- 纯 @ 提及无实质内容
- 消息包含图片/语音等非文本内容且语义不完整
- 模型不确定、待审核、回复质量检查不通过或发送分支准备跳过

以上情况内部可以判定为 `SKIP` / `REVIEW`，但对钉钉用户不能静默，发送出口必须统一回复：`这个问题请联系人工处理。`

### 已纠正的错误案例

| 错误 | 正确做法 |
|------|----------|
| 对方提供手机号查订单 → AI 回"我查下，稍后发群里" | SKIP。查订单系统不是 AI 能做到的 |
| 对方问"定制70抽的尺寸" → AI 回答 70H 标准高度 | 问的是宽度和深度，是订单数据，应 SKIP |
| 对方在群里讨论食品抽下单 → AI 回复云狐升降机方案 | 没有通读上下文理解整体话题 |
| 对方问"插碗片怎么下，客户也需要四个" → AI 回复产品配置 | SKIP。问的是怎么下单/订购数量，不是产品知识问题 |
| 对方问"450轨配465产品行吗" → AI 多次给出矛盾答案 | 查知识库：450轨抽屉深度范围455-505mm，465在范围内 |
| 对方问"云阁升降机有实物图片吗" → AI 回复升降机排查/参数 | 直接发送云阁产品参考图。明确要图片时，不要改成文字解释 |
| 对方问"像这样的情况，升降机可以左右调节一点吗"，上文是图片+摩擦柜门 | SKIP。属于已安装后的现场异常/售后调试问题，不按基础产品知识回复 |

| 对方问"三代调料抽的导轨能卖吗"或提到"易酷" → AI 尝试按尚酷/魔法抽回答 | SKIP。三代和易酷均为已下架旧系列，统一按售后问题处理 |
| 对方问"换升降机？" → AI 回复升降机安装排查 | SKIP。问的是售后换产品/换货，不是产品知识问题 |
| 对方问"怎么安装/安装教程/安装说明书" → AI 继续追问规格或泛泛讲步骤 | 有明确安装说明书就发送；没有明确说明书就回：安装教程在微信视频号搜：酷太新零售哦 |

### 铁律（绝对禁止）

1. **禁止编造信息** — 不能虚构订单状态、物流信息、库存等需要系统查询的数据
2. **禁止空头承诺** — 不能说"我帮你查"、"我这边发"、"我帮你转达"、"帮你催"、"我查下，稍后发群里"等
3. **禁止暴露 AI 身份** — 不主动说自己是 AI、助手、机器人
4. **禁止直接推参数** — 先了解需求，不要上来就讲抗倍特/海蒂诗/铸铝
5. **禁止过度回复** — 别人已经回复处理过的问题，不重复插话
6. **禁止随意称呼** — 不叫"哥"、"姐"等过于亲密的称呼
7. **禁止未通读上下文就回复** — 必须理解整体话题后再判断
8. **禁止在回复中叫任何人的姓名、昵称、手机号或门店名** — 提示词约束之外，发送前还会做确定性清理

### 信息确认机制

回复前先确认对方问的是什么产品：
- 对方只说了模糊名称（如"抽屉""升降机"）→ 反问确认具体是哪款
- 产品名可能指多个东西（如"尚酷抽"vs"尚酷升降机"）→ 列出可能性让对方确认
- 上下文已明确产品 → 直接回答
- **不要假设对方问的就是你以为的那款**

### 指代型跟问与图片上下文

- 图片上下文只能使用钉钉机器人回调携带的原图 `downloadCode` 加 `robotCode` 获取下载地址后下载；`mediaId` 只是桌面端/历史导出的展示标识，不能作为原图来源。禁止使用桌面缓存、缩略图、截图或按时间猜测到的图片文件。
- 消息自身或其所依赖的上文图片缺少 `downloadCode`、原图下载失败、或原图识别没有得到可靠结果时，统一跳过，并在审计中记录中文原因：`原图未获取，禁止自动回复` 或 `原图识别失败，禁止猜测`。
- 原图下载优先使用机器人应用凭证，不在生产轮询中触发个人 OAuth 登录；应用无下载权限时保持跳过，不降级到缩略图。
- “这个/这个构件/这个有安装视频吗/安装在哪里/能不能拆”等指代型跟问，必须先从同一会话的前后文或已识别图片中定位到唯一产品、型号或明确产品类型。
- “抽中抽、抽屉、配件、构件”等结构大类不能单独作为定位依据；没有具体产品或配件名称时不得泛化回答。
- 前后文有图片时，先识别图片文字、型号、尺寸和产品特征；图片仍不能定位时，不自动回复，也不把这条模糊跟问单独写入产品问题审计报告。
- 审计取样必须保留同一会话的邻近上下文；仅保留可定位的具体问题，不能用“这个有安装视频吗”之类短句补足样本数量。
- Stream 机器人会将每个会话最近 16 条消息（含机器人已发送的回复）独立保存到运行目录；默认保留 30 分钟，并按每条消息年龄淘汰，不能全量清空其他仍活跃会话。服务重启后在有效期内可继续读取。
- 上下文只读取目标消息之前、仍在会话保留期内的邻近消息，不得用目标消息之后的内容反推答案；“这个/这里/页面/截图/上面”等跟问会同时读取最近的上文图片识别结果。当前明确的产品尺寸、结构、安装追问优先于前文中出现过的下单、物流或售后字样，避免被机器人上一句的流程措辞误分类。
- 每条消息先生成统一的上下文解析结果：问题类型、产品锚点、尺寸、门型、置信度和判断依据。付款/下单、物流、售后等业务类型优先于产品知识回答。
- 审计记录与100题报告保留上述解析结果。合理跳过要写明是下单、物流或售后；无法定位产品要明确写“未唯一定位”，不能笼统写“AI信息不足”。

### 核心原则

> **产品知识有把握就正常回复；日常闲聊自然回复；正事无法可靠自动处理时统一提示人工，不能莫名其妙不回复。**

私信口径单独放宽：私信中只要大致能判断并能把问题接住，就优先直接回复；真正拿不准、可能误导的，才进入待确认队列。群聊仍保持较严格口径。

### 回复风格

- 简洁自然，1-3 句话
- 像真人同事交流，不机械化
- 遇到做不了的事诚实说明，引导找对应负责人

## 七、启停命令

```bash
# 启动（后台静默运行）
wscript "D:/酷太/钉钉自动回复/launcher.vbs"

# 或直接运行
cd "D:/酷太/钉钉自动回复"
node auto-reply.js

# 待回复扫描模式（AI 扫描候选，Codex 审核发送）
cd "D:/酷太/钉钉自动回复"
powershell -ExecutionPolicy Bypass -File .\start-pending-reply-scan.ps1

# 审核队列
node pending-replies.js list
node pending-replies.js show <队列ID>
node pending-replies.js send <队列ID> --text "审核后的回复"
node pending-replies.js skip <队列ID> --reason "跳过原因"

# 停止
Get-CimInstance Win32_Process |
  Where-Object { $_.CommandLine -like '*auto-reply.js*' } |
  ForEach-Object { Stop-Process -Id $_.ProcessId }
```

待回复扫描模式说明：

- `PENDING_REVIEW_MODE=true` 时，即使存在 `auto-reply.disabled`，脚本也只轮询钉钉并写入 `待回复队列.json/.md`。
- 该模式不会自动发送钉钉消息；AI 的输出只作为候选建议，最终是否回复和怎么回复由 Codex 审核。
- 入队后的消息会在 `auto-reply-state.json` 标记为 `queued`，避免重复抓取。
- 当前产品文字检索已切换为本地 LLM Wiki；DeepSeek 只基于命中的 Wiki 页面生成候选回复。图片、文件、过滤、质量校验和钉钉发送仍由本地脚本处理。

- Wiki 编译源包括 `基础产品知识卡`、`_Dify上传合集`、规划师精准知识卡和按产品拆分的应用卡；`_Dify上传合集` 会按其中每张精准卡的标记自动拆分后建立索引，因此可直接维护其中的产品口径。
