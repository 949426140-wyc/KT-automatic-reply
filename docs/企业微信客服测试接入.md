# 企业微信客服测试接入

本项目使用“微信客服 API”，不是企业微信群机器人。消息路径为：企业微信事件回调 → `kf/sync_msg` 拉取原始消息 → LLM Wiki/DeepSeek 生成内部结果 → （明确开启后）`kf/send_msg` 发给客户。

## 当前安全默认值

- `WECOM_KF_ENABLED=false`：不拉取消息。
- `WECOM_KF_AUTO_SEND=false`：即使收到消息，也不向客户发送。
- 仅 `origin=3` 的客户主动消息进入产品问答；人工消息、系统事件一律跳过。
- 目前企业微信客服仅支持文本进入问答。图片在“原图下载和视觉识别”接通前只记录为跳过，绝不使用缩略图推断答案。

## 必填配置

在部署用 `.env` 中填入以下值（不要提交到 Git）：

```dotenv
WECOM_KF_ENABLED=true
WECOM_KF_AUTO_SEND=false
WECOM_KF_CORP_ID=企业微信CorpID
WECOM_KF_SECRET=新建“酷太产品助手”的Secret
WECOM_KF_CALLBACK_TOKEN=企业微信回调页面自行生成的Token
WECOM_KF_CALLBACK_AES_KEY=企业微信回调页面自行生成的EncodingAESKey
```

`WECOM_KF_AUTO_SEND` 仅在内部模拟审计、人工试问和客服账号确认均完成后才可改为 `true`。

## 企业微信后台设置顺序

1. 使用测试自建应用“酷太产品助手”，不要改动正在使用的旧应用或正式客服账号。
2. 在微信客服的“通过 API 管理会话消息 → 企业内部开发”中，把测试客服账号授权给“酷太产品助手”。
3. 在“酷太产品助手 → 接收消息 → 设置 API 接收”配置：
   - URL：`https://你的公网域名/wecom/kf/callback`
   - Token：与 `WECOM_KF_CALLBACK_TOKEN` 相同
   - EncodingAESKey：与 `WECOM_KF_CALLBACK_AES_KEY` 相同
4. 保存时企业微信会请求 `GET /wecom/kf/callback` 验证；成功后才会推送 `kf_msg_or_event`。
5. 先通过测试客服链接发送一条产品文本问题。系统应只在审计日志中产生 `wecom_kf_inbound` 和 `wecom_kf_decision`，不对客户发消息。
6. 人工确认审计答案后，再单独评估是否开启自动发送。

## 接口与审计

- 健康检查：`GET /` 的 `wecomKf` 字段会显示开关、回调和 API 配置状态，不返回任何密钥。
- 回调地址：`GET/POST /wecom/kf/callback`
- 审计：`data/bot-reply-audit.jsonl`。企业微信跳过原因使用中文。

## 公网 HTTPS 前置条件

企业微信不能访问 `localhost`、局域网 IP 或 Docker 内部地址。必须提供稳定的公网 HTTPS 域名并将请求转发到本机 `3000` 端口；完成前不要在企业微信后台保存回调。
